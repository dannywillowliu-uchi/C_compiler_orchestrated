# Checkpoint 1: C Compiler from Scratch via Autonomous AI Agents

## What We Did

Built a functioning C compiler from scratch in Python using autonomous AI agents orchestrated by [mission-control](https://github.com/dannywillowliu-uchi/autonomous-development). No human wrote any compiler code -- every line was generated by AI workers coordinated by an automated planner.

### The Process

1. **Created the project** -- empty repo with a stub `__main__.py`, a `pyproject.toml`, and a `mission-control.toml` config describing the objective: "Build a functioning C compiler that takes C source and produces x86-64 assembly."

2. **Launched mission-control** -- the orchestrator reads the objective, invokes a planner LLM to decompose work into parallel units, dispatches units to 4 worker agents (each a Claude Code session in an isolated git worktree), merges results via a green-branch strategy, and replans each epoch based on the evolving codebase.

3. **Monitored and fixed infrastructure issues** -- during the run, we identified and fixed:
   - A planner prompt issue where acceptance criteria referenced `/tmp/` absolute paths, causing units to be rejected by the path security validator
   - A validator bug where it compared paths against the worker workspace instead of the actual project root
   - Merge conflicts on shared files (parser reimplementing lexer/AST independently), handled by the retry/fixup system

4. **Three mission runs** over ~5 hours of wall time produced the full compiler.

### How mission-control Works

```
Objective -> Planner LLM -> Work Units (topologically sorted)
                                |
                    +-----------+-----------+
                    |           |           |
                Worker 1   Worker 2   Worker 3   (parallel, isolated worktrees)
                    |           |           |
                    +-----------+-----------+
                                |
                         Merge to mc/green -> Push to main
                                |
                         Replan next epoch (survey codebase, identify gaps)
```

Each epoch: the planner surveys the codebase, identifies what's missing, proposes 3-4 units with file-level isolation (no two units touch the same file), workers execute in parallel, results merge, and the cycle repeats.

## What the Compiler Can Do

### Full Pipeline
```
C source -> Preprocessor -> Lexer -> Parser -> Semantic Analysis -> IR Generation -> IR Optimization -> Code Generation -> Peephole Optimization -> x86-64 Assembly
```

Invoked as: `PYTHONPATH=src python -m compiler input.c -o output.s`

### Language Features Supported

**Types:**
- `int`, `char`, `void`, `float`/`double` (SSE codegen)
- `unsigned` arithmetic
- Pointers (including pointer arithmetic, dereference, address-of)
- Arrays (with pointer decay)
- Structs (with member access `.` and `->`, proper alignment/padding)
- Unions
- Enums
- Typedefs
- Type qualifiers (`const`, `volatile`, `static`, `extern`)
- Cast expressions

**Expressions:**
- Arithmetic: `+`, `-`, `*`, `/`, `%`
- Comparison: `==`, `!=`, `<`, `>`, `<=`, `>=`
- Logical: `&&`, `||`, `!` (with short-circuit evaluation)
- Bitwise: `&`, `|`, `^`, `~`, `<<`, `>>`
- Assignment: `=`, `+=`, `-=`, `*=`, `/=`, `%=`, `&=`, `|=`, `^=`
- Prefix/postfix `++`/`--`
- Ternary `? :`
- `sizeof`
- Comma operator
- Function calls (direct and via function pointers)
- Array subscript `[]`
- Struct/union member access `.` and `->`
- String literals with escape sequences
- Adjacent string literal concatenation

**Statements:**
- `if`/`else`
- `while`, `for`, `do`-`while`
- `switch`/`case`/`default`
- `break`, `continue`
- `return`
- Compound statements (blocks)
- Multi-variable declaration lists

**Declarations:**
- Functions (with parameters, System V ABI calling convention)
- Global variables (with `.data`/`.bss` sections)
- Local variables
- Struct/union definitions
- Brace initializer lists for arrays and structs
- Storage classes (`static`, `extern`)

**Preprocessor:**
- `#define` (object-like and function-like macros)
- `#ifdef`, `#ifndef`, `#if`, `#elif`, `#else`, `#endif`
- `#include`
- `#undef`
- `#error`, `#warning`
- `#line`
- `#pragma once`
- Token pasting (`##`) and stringification (`#`)

### Compiler Internals

**Optimization passes:**
- Constant folding
- Copy propagation
- Dead code elimination (DCE)
- Common subexpression elimination (CSE)
- Loop-invariant code motion (LICM)
- Strength reduction
- Jump threading
- Unreachable code elimination

**Peephole optimizer:**
- Redundant mov elimination
- Mov chain simplification
- LEA optimization
- Push/pop elimination
- Dead store removal
- Jump-to-next elimination
- Address-mode folding
- Redundant comparison removal

**Analysis infrastructure:**
- Control flow graph (CFG) construction
- Liveness analysis
- Graph-coloring register allocator

**Code generation:**
- x86-64 AT&T syntax
- System V AMD64 ABI (args in `%rdi`, `%rsi`, `%rdx`, `%rcx`, `%r8`, `%r9`; return in `%rax`)
- SSE instructions for floating-point
- Proper stack frame management
- Global/static data sections
- String literal pooling

## Example

Input:
```c
int fibonacci(int n) {
    if (n <= 1) return n;
    return fibonacci(n - 1) + fibonacci(n - 2);
}

int main() {
    int result = fibonacci(10);
    return result;
}
```

Output (x86-64 assembly):
```asm
.section .text
.globl fibonacci
.type fibonacci, @function
fibonacci:
    pushq %rbp
    movq %rsp, %rbp
    subq $96, %rsp
    movq %rdi, -8(%rbp)
    movq -8(%rbp), %rax
    movq $1, %rcx
    cmpq %rcx, %rax
    setle %al
    movzbq %al, %rax
    cmpq $0, %rax
    jne if_then0
    jmp if_end2
if_then0:
    movq -8(%rbp), %rax
    movq %rbp, %rsp
    popq %rbp
    ret
if_end2:
    movq -8(%rbp), %rax
    movq $1, %rcx
    subq %rcx, %rax
    movq %rdi, %rax
    call fibonacci
    movq %rax, -64(%rbp)
    movq -8(%rbp), %rax
    movq $2, %rcx
    subq %rcx, %rax
    movq %rdi, %rax
    call fibonacci
    movq %rax, -88(%rbp)
    movq -64(%rbp), %rax
    addq -88(%rbp), %rax
    movq %rbp, %rsp
    popq %rbp
    ret
```

## Known Gaps (for Checkpoint 2)

- No linker integration (assembly output only, not executable)
- No `long`, `short`, `signed` type modifiers beyond `unsigned`
- No variadic functions (`...`)
- No `goto`/labels
- No bitfields in structs
- No complex initializers for globals
- No `_Bool` / `<stdbool.h>`
- Register allocator integrated but assembly still uses many stack spills
- No `#include` path search (preprocessor expects literal paths)
